The policer recipe (full derivation + the interface-ref trap: docs/07 §6.1):

```
$ docker exec -i clab-a2a-srl1 sr_cli <<'EOF'
enter candidate
set / qos policer-templates policer-template police-50m policer 1 peak-rate-kbps 50000 …
set / qos interfaces interface ethernet-1/1.0 interface-ref interface ethernet-1/1
set / qos interfaces interface ethernet-1/1.0 interface-ref subinterface 0
set / qos interfaces interface ethernet-1/1.0 input policer-templates policer-template police-50m
commit now
EOF
All changes have been committed.
```

The discovery (docs/07 §6.2, ADR-006): committed ≠ enforced in the container —
UDP 100M still passed 99.5 Mbps / 0 % loss; policer state reads `peak-rate-kbps 0`;
ACL drop enforced, ACL rate-limit not; tc-ingress XDP-shadowed; **tc egress works**.

After — policer committed **+ one shim tick** (`netlab/mirror-policer-to-tc.sh`):

```
$ ./netlab/mirror-policer-to-tc.sh

$ docker exec clab-a2a-hostA iperf3 -c 10.10.2.10 -t 10
[  5]   0.00-10.06  sec  57.2 MBytes  47.7 Mbits/sec   receiver   ← THE PLATEAU
$ docker exec clab-a2a-hostA iperf3 -c 10.10.2.10 -u -b 100M
```

## Surprises / deviations from the plan

- **The plan's premise ("SR Linux policer → plateau") is false in containers**: the
  XDP datapath accepts and then ignores QoS policers (state zeroed, traffic untouched).
  Resolved by ADR-006: the gNMI config stays the source of truth; lab infrastructure
  mirrors it to `tc tbf` inside srl1's netns. Verified end-to-end both ways
  (attach → 48; detach → ceiling). This is very likely what the deleted
  `srl-gnmi-bandwidth-poc` repo on this machine was wrestling with.
- **The misleading error of the day:** `"Attachment of policier-templates, not permitted
  for interfaces"` (sic) does not mean "platform forbids policers" — the YANG `must`
  wants `interface-ref { interface, subinterface }` populated alongside the list key.
  Found by grepping the YANG inside the image. Two platform-type detours (D1: no QoS
  tree at all in container; 7250 IXR-6e: different QoS model, node too heavy for this
  machine) were wrong turns caused by trusting that error text.
- **MTU ambush**: hosts came up at MTU 9500, srl1 routes at IP-MTU 1500 — pings passed,
  TCP moved 559 Kbit/s with 100 % receiver starvation. `mtu 1500` now baked into the
  topology exec lines.
- ADR numbering shifted: the lab-datapath decision took ADR-006; the telemetry delivery
  model (Phase 3) becomes ADR-007 (docs/01 updated).
