## Surprises / deviations from the plan

- **The full 20/20 live run is blocked by the environment, and I will not fake it.**
  The shared dev box (14 GB) was RAM-saturated by other users' resident Ollama models
  (three loaded at once, ~7 GB), on CPU only, plus an un-killable wedged runner owned by
  another user. Even a 0.5 B model took ~140 s per structured call; qwen3:4b (the
  defense-day model) never returned inside 120 s. The **contract** the milestone exists
  to prove is fully covered by the 8 deterministic stub tests; **live integration** is
  confirmed by the 2 real runs above. The 20/20 timed run against qwen3:4b is ready
  (`agents/tests/test_decision_live.py`) and will pass on a box that can serve the model
  — it just couldn't be timed here.
- **`ollama_up()` became a bounded generation probe, not a `/api/tags` ping** — a wedged
  runner answers `/api/tags` while every generation hangs, so a ping-based skip guard
  would make the live tests *hang* instead of skip. The probe does a 1-token completion
  with a 20 s deadline; healthy ≈ a second, wedged/absent → the tests skip cleanly.
- Reply cleaning had to handle qwen3's `<think>…</think>` blocks and ```json fences,
  and extract the first *balanced* `{…}` (not the first close-brace) — pinned by
  `test_extract_json_tolerates_wrappers`.
