## Surprises / deviations from the plan

- **The adversarial review (37-agent workflow) caught a shipped-in-all-but-name bug:**
  `vm.createDir(path, false)` maps to `fs::create_dir`, which *errors* when the directory
  exists — so every `just deploy-local` after the first would have failed
  (`File exists (os error 17)`), directly contradicting docs/03 §2.4's "regenerate at
  will". A verifier reproduced it empirically before I ever hit it. Fix: `recursive=true`
  (= `create_dir_all`, a genuine no-op); verified by deploying twice in a row. The same
  review caught nine staleness/wording issues (plan-of-record paths, a renamed script's
  docstring, a garbled placeholder line in the lab) — all fixed pre-merge.
- **Foundry hard-denies fs writes above its own root** — the plan's `deployments/anvil.json`
  at repo root is impossible (`fs_permissions` with `../deployments` is rejected at
  runtime). Artifact moved to `contracts/deployments/anvil.json`; docs/03 §2.4 records
  the shape and location. `vm.writeJson` also refuses to create directories —
  `vm.createDir` first.
- **`script/Deploy.s.sol` was the M1.1 Counter lab artifact** — renamed to
  `DeployCounter.s.sol` (EXPLORE.md references updated) so the real deploy could take the
  canonical name the plan promised.
- Re-revoking is *deliberately idempotent* (flag re-set, event re-fired) — parity with
  `FakeChain.revoke`, and harmless downstream because teardown is idempotent (rule 8).
  Pinned by `test_revokeTwiceIsIdempotent`.
- `tokenURI` omits `params` (ABI blob ≠ JSON-friendly; readers decode from
  `entitlements(id)`) but includes `resourceId`/`termsHash` — the URI alone identifies
  what was sold.
