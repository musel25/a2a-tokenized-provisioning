## Explain-back answers (write them, two sentences each is enough)

1. Why must validation live at the boundary rather than inside each package?
   > The boundary is the *seam where untrusted bytes enter a package* — the point where an
   > A2A payload or a chain read is deserialized into a model, not the `interfaces` module
   > itself. Validating once there lets all interior code trust the data, so no package
   > re-checks defensively and no two packages drift on what "valid" means (docs/04 §2:
   > "validation is an event at a boundary, not a property of a type").
2. What's the difference between a Protocol and a base class, and why does the
   controller's testability depend on it?
   > A base class is *nominal* — a class only qualifies by explicitly inheriting it; a
   > Protocol is *structural* — any class with the right methods fits, no inheritance and
   > no import. The controller's domain depends only on the `EntitlementReader` /
   > `NetworkProvisioner` Protocols, so a ~30-line fake and the real web3/pygnmi adapter are
   > interchangeable and the same contract-test suite runs against both (rules 4, 7) — with a
   > base class the mock would have to import and subclass the real adapter, recoupling them.

## Surprises / deviations from the plan

- **Doc bug found & fixed:** docs/03 §2.1 said the `Offer` struct has "eleven fields"
  but §1.4 lists **twelve** (`provider, consumer, serviceType, resourceId, params,
  startTime, endTime, paymentToken, price, validUntil, salt, termsHash`). Transcribed
  twelve (§1.4 JSON is authoritative); corrected §2.1.
- **pydantic everywhere (design decision).** docs/03 showed the port-side types
  (`EntitlementView`, `ResolvedPath/Node`, `ApplyResult`) as `@dataclass(frozen=True)`.
  We used **frozen pydantic models** for all shapes — one validation system, uniform
  JSON round-trip, immutability preserved. Fields are identical, so this is a
  representation choice, not a shape change: `v` stays `0`. Synced the four docs/03
  snippets to pydantic in this commit (CLAUDE.md rule 3).
- **Field-naming split is real, not a mistake:** A2A payloads (`ServiceNeed`) are
  snake_case on the wire; the `Offer` struct is camelCase (mirrors Solidity). Handled
  with a camelCase alias generator on `Offer` only.
- **Validation depth is honest about its limits:** addresses are checked by *pattern*
  (`0x` + 40 hex), not EIP-55 checksum — checksum needs keccak, which belongs to
  chainmcp (rule 2). `bytes` fields on `EntitlementView` round-trip through JSON as hex
  via `ser/val_json_bytes="hex"`.
- `terms_hash`, `salt`, and the `signature` in fixtures are syntactically-valid
  placeholders; real keccak/signing arrives in M1.5.
