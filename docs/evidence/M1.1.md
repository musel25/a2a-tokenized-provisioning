## Explain-back answers (write them, two sentences each is enough)

1. Transaction vs call — which one will `fulfill` be, which one will the controller's
   reads be, and why does that matter for cost and trust?
   > `fulfill` is a **transaction**: it costs gas, must be signed with a key, and mutates
   > shared state (marks the offer salt consumed, moves TOK consumer→provider, mints the
   > NFT) — atomically. The controller's reads (`ownerOf`, the revoked flag, the validity
   > window) are **calls**: free, no key, read-only. **Cost:** only the state-changing op
   > pays gas. **Trust:** a call reads the chain's *consensus state* — the controller
   > isn't trusting a provider's database, it reads the same on-chain truth everyone sees;
   > a transaction *requires* a signature precisely because it spends money and changes
   > ownership, so identity must be proven to mutate shared state.
2. What is an event *for*, in one sentence, given the controller's `Revoked` watcher?
   > An event is a cheap, watchable notification that *something happened* — emitted by a
   > state-changing transaction and written to that transaction's log (in the block's
   > receipts: on-chain and consensus-secured via `receiptsRoot`, but not in contract
   > storage and not readable by other contracts). It lets off-chain code react the
   > instant `revoke(7)` mines — the node *pushes* `Revoked(7)` to the controller's
   > subscribed watcher (with who/what indexed for filtering, plus full history),
   > instead of the controller polling storage on a loop and risking missing a brief
   > revoked window.
